## 1. 缺陷验证

### 缺陷 1：模板策略同质化 — 部分确认

**已确认的同质性：**
- `generate_stream()` 无模板 override（base.py:573-712 唯一实现）
- 问题预算硬编码 40/25/20/15（base.py:759-787）
- 评分权重全局固定 30/25/25/20（protocol.py:42-48）
- simulation `store_ratio=0.7` 在所有模板上接近最优
- strategic 准确率跨模板 85-100%（紧密聚集）

**已存在的差异化（未被利用）：**
- 关系类型：company(2), research(3), city(2), hospital(2), sport(2), movie(2)
- list_float 模式：seasonal/logistic/monotonic/periodic（各模板不同）
- 属性分布：company 13/23 float vs research 13/23 int
- text/enum 数量：hospital 3 text vs company 1 text

**结论**：差异化存在于**答题层**（不同计算类型），但不影响**存储决策层**。一个通用 "存 70%、处理 correction" 策略在所有模板上接近最优。缺陷成立。

### 缺陷 2：推理机械化 — 确认

**代码证据：**
- simulation.py:212：comprehension 用 GT-against-GT 验证（"assumes perfect computation if data is available"）
- 18 种题型全部是确定性运算：argmax/sum/avg/filter/divide/count/path-traversal
- 准确率与存储率线性相关：perfect=100%, strategic=65%, naive=19%（零推理成分）
- system prompt 给出精确答题格式（stream_agent.py:75-123）
- 2% float 容差（validators.py:49）

**结论**：comprehension axis 实际测量的是"存储广度 × 基础算术"，不是推理能力。缺陷成立。

### 缺陷 3：真实场景脱节 — 部分确认，需要 nuance

**已确认的脱节：**
- selective redaction 强制工具使用（eval_task.py:276 删除历史上下文）
- 信息量完全已知（total_entities 提前暴露）
- 批次式处理（非连续流）

**被否定的担忧：**
- "文档结构化" — 实际用叙述体渲染，含干扰项（base.py:360-413），非显式结构
- "存储全有或全无" — detect_stored_entities 只需 name+1attr（questions_advanced.py:240-248），模型可压缩存储
- "负迁移必然发生" — selective redaction 在 context 被裁剪后是合理的（否则模型无法回溯），不是强制性反模式
- "奖励信号不真实" — 已有 binary + shaped + episode-level 三层（training.py:531/481/564）

**结论**：主要问题是信息量预知和批次式处理，而非 selective redaction 本身。selective redaction 是合理的设计选择（与 MemGPT 的 context paging 类似）。缺陷部分成立。

## 2. 前沿研究摘要

| 系统/基准 | 核心贡献 | 与 MemoryGym 的差异 |
|-----------|---------|-------------------|
| **MemGPT/Letta** | OS-inspired 分层记忆（main/recall/archival） | 无存储预算，测检索质量不测存储决策 |
| **LoCoMo** | 跨会话长期对话记忆，5 种问题类型含时间推理 | 无预算/信息过载/correction |
| **MemoryAgentBench** | 4 能力：检索/在线学习/长程理解/冲突解决 | conflict resolution ≈ MemoryGym maintenance，但无预算 |
| **LongMemEval** | 500 题 × 2 scale，indexing/retrieval/reading 分析框架 | 无预算，测 pipeline 质量 |
| **Mem-alpha** | RL 训练记忆策略，30k→400k+ 泛化 | 无硬预算，隐式学习效率 |
| **AMemGym** | 交互式在策略评测，发现 "Reuse Bias" | 验证了 MemoryGym 的 seed 生成策略 |
| **AMA-Bench** | agent 环境交互轨迹记忆，因果建模 | 强调真实 agent 场景的非对话特性 |

**MemoryGym 独特价值**：唯一同时具备 (1) 硬预算约束 (2) 抗博弈验证 (3) RL 训练环境 的基准。

**可借鉴的设计**：
- LongMemEval 的 indexing/retrieval/reading 诊断框架
- MemoryAgentBench 的在线学习能力测试
- Mem-alpha 的 RL 训练信号设计（短→长泛化）
- AMA-Bench 的因果追踪（不只是 "correction happened"）

## 3. 升级方案

### 方案 A：模板策略差异化

**核心思路**：让不同模板的最优存储策略本质不同，使通用策略失效。

#### A1. 模板特定的事件流结构

每个模板 override `generate_stream()` 引入领域特定模式：

| 模板 | 事件流特性 | 最优策略差异 |
|------|-----------|-------------|
| company | **级联更新**：收购导致多实体同时变化 | 需要关联存储，不能独立决策 |
| research | **引用网络**：新论文引用旧论文，更新引用计数 | 需要维护引用图，优先存高 h-index |
| hospital | **高频状态变化**：患者状态每轮可能变化 | 需要预留大量更新预算 |
| city | **稳定少变**：城市数据很少修正 | 可以多存、少预留更新预算 |
| sport | **赛季结构**：赛季中期有大量统计更新 | 需要批量更新策略 |
| movie | **票房衰减**：首周高、后续低，需跟踪趋势 | 需要时间序列存储 |

**实现**：
- 各模板 override `generate_stream()`
- 引入 `correction_rate` 属性（hospital 高、city 低）
- simulation 新增 "template_expert" 策略：了解领域特性的专家策略
- 验证标准：template_expert > generic_strategic 在该模板上

#### A2. 模板特定的问题分布

允许模板自定义问题类型权重：

```python
class CompanyWorld(WorldTemplate):
    question_weights = {
        "retrieval": 0.30,      # 减少（不是核心测试）
        "comprehension": 0.35,  # 增加（财务推理更重要）
        "update": 0.20,
        "abstention": 0.15,
    }
```

**约束**：所有模板仍输出标准 4 轴分数，只是题目比例不同。

#### A3. 实体重要性不均匀

当前所有实体被提问的概率相等。改为：

- 部分实体是 "关键实体"（被更多问题引用）
- 关键实体通过关系图度数、属性极值等特征可识别
- 智能 agent 应识别并优先存储关键实体
- 验证标准：priority_strategic（识别关键实体）>> random_strategic（随机存储）

**simulation 新策略**：`importance_aware`（根据实体特征决定优先级）vs `uniform`（等概率存储）

### 方案 B：推理深度提升

**原则**：新题型必须 (1) 可确定性评分 (2) guesser=0% (3) 需要存储+推理

#### B1. 反事实问题（counterfactual）

"如果修正没发生，Entity X 的 attr 值是多少？"

- **GT 计算**：correction.old_value（已存储在 Correction 对象中）
- **需要**：agent 同时记住修正前和修正后的值
- **guesser baseline**：0%（需要知道旧值）
- **当前系统影响**：不需要改问题框架，只需新增一个 `_gq_counterfactual()` 方法
- **simulation**：perfect 策略存储 old_value+new_value → 100%

#### B2. 多约束过滤（multi_constraint）

"满足 revenue > $50M AND employees < 1000 AND sector = 'Technology' 的实体有几个？"

- **GT 计算**：`len([e for e in entities if e.revenue > 50M and e.employees < 1000 and e.category == 'Tech'])`
- **需要**：存储多个属性并组合过滤
- **guesser baseline**：接近 0%（组合空间大）
- **当前系统影响**：新增 `_gq_multi_constraint()` 方法
- **变体**：2-constraint（中等），3-constraint（困难）

#### B3. 时序推理（temporal_order）

"Entity A 的修正和 Entity B 的修正，哪个先发生？"

- **GT 计算**：从 correction 事件的 batch 顺序推导
- **需要**：记住修正发生的相对顺序
- **guesser baseline**：50%（二选一）→ 可扩展为 3+ 实体排序降低 baseline
- **当前系统影响**：需要在 stream 中暴露时间标记（batch index 或相对时间）

#### B4. 推断缺失值（inference）

"Entity A (category=X) 的 attr1 已知。同类别的平均 attr1 是 Y。Entity B (category=X) 的 attr1 未被存储。估算 B 的 attr1。"

- **GT 计算**：类别平均值（从 world 数据计算）
- **需要**：理解类别统计并推断
- **guesser baseline**：0%（需要类别知识）
- **注意**：需要较宽容差（10-20%），因为推断本身不精确

**推荐优先级**：B1 > B2 > B3 > B4（B1 最容易实现且不可作弊，B4 需要新的验证逻辑）

### 方案 C：真实性增强

#### C1. 隐藏信息量（不暴露 total_entities）

当前 budget context 显示 "Entities seen: 23/60"。改为只显示已见数量，不显示总数：

"Entities seen: 23. Budget: 12/30 writes remaining."

- **影响**：模型不知道还有多少实体来，必须更保守地分配预算
- **验证**：strategic 准确率可能下降（预算分配更难），但 perfect 仍为 100%

#### C2. 可选长上下文模式

新增 eval 模式：不做 selective redaction，保留完整历史。

- 测量"长上下文 vs 工具辅助记忆"的差异
- 作为额外数据点，不替换现有评测
- 实现：eval_task.py 新增 `--no-redaction` flag

#### C3. 非结构化信息混入

在 entity 文档之间穿插噪声文本（模拟真实信息流的混乱）：

- "公司内部邮件" / "会议纪要摘要" / "新闻摘要"
- 包含部分实体名但无完整属性（干扰存储决策）
- 不计入写入预算（免费看到但需要过滤）

## 5. 实施计划

按影响和可行性排序，拆分为独立 Phase：

### Phase 30 — 反事实推理 + 多约束过滤题型（推理深度 B1+B2）
- 新增 `_gq_counterfactual()` 和 `_gq_multi_constraint()`
- 更新 simulation：perfect 存 old_value → counterfactual 100%
- 验证：guesser=0%, smart_guesser<5%
- 预计：+2 题型，questions_advanced.py 增加 ~100 行

### Phase 31 — 模板事件流差异化（策略差异化 A1）
- 各模板 override `generate_stream()`
- 引入模板特定的 correction_rate 和事件模式
- 新增 simulation 策略：template_expert
- 验证：template_expert > generic_strategic 在对应模板上
- 预计：6 个模板文件各增加 50-100 行

### Phase 32 — 实体重要性分化 + 问题分布定制（A2+A3）
- 实体重要性信号（关系度数、极值）
- 模板特定的 question_weights
- 验证：importance_aware > uniform_strategic
- 预计：base.py + 6 模板文件

### Phase 33 — 信息隐藏 + 噪声注入（真实性 C1+C3）
- 隐藏 total_entities
- 非结构化噪声文本穿插
- 验证：所有 simulation 不变量仍成立
- 预计：eval_task.py + stream_agent.py + base.py

### Phase 34 — 长上下文评测模式（C2）
- `--no-redaction` flag
- 对比数据收集
- 预计：eval_task.py 小改

### Phase 35 — V2 评测数据收集
- 用升级后系统跑全模型 eval
- 对比 V1 数据，验证改进有效性

## 6. 风险和缓解

| 风险 | 缓解 |
|------|------|
| 模板 override generate_stream() 导致代码膨胀 | 共享基础逻辑，只 override 差异部分 |
| 新题型破坏 simulation 不变量 | 每个 Phase 必须通过全量 simulation |
| 向后不兼容 | 新功能默认关闭，通过 tier 配置开启 |
| 实体重要性可被逆向工程 | 重要性信号来自世界状态（关系图），非外部标记 |
| 反事实题难度过高导致所有模型 0% | 从 1-2 题开始，渐进增加 |
